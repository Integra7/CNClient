# Техническое задание на доработки бэкенда CosaNostra

**Версия:** 1.0  
**Дата:** 2025  
**Цель:** обеспечить работу веб-клиента (логин, регистрация, чаты, сообщения) через WebSocket API.

---

## 1. Область применения

- **Система:** бэкенд мессенджера CosaNostra (Kotlin, Ktor, WebSocket, PostgreSQL).
- **Клиент:** веб-приложение в репозитории CNClient; взаимодействие только по WebSocket (JSON).
- **Токен в MVP:** UUID пользователя (передаётся в query-параметре `token` при подключении к WebSocket).

---

## 2. Приоритеты и порядок работ

| Приоритет | Задача | Зависимости |
|-----------|--------|-------------|
| P0 (критично) | Подключение с пустым токеном для входа/регистрации | — |
| P0 | Логин по username + password | P0 подключение |
| P0 | Список чатов `get_chats` с именем для личных чатов | — |

## 3. Задачи (подробно)

### 3.1. Подключение с пустым токеном (P0)

**Цель:** дать клиенту возможность войти или зарегистрироваться до выдачи UUID.

**Требования:**

1. Разрешить установку WebSocket-соединения по адресу вида:
   - `ws://host:8080/chat?token=`
   - или при отсутствии параметра `token`.
2. Для сессий с пустым/отсутствующим токеном:
   - принимать **только** сообщения с `type`: `login`, `create_user`;
   - на остальные типы сообщений — не выполнять действие (или отвечать `error`).
3. После успешного ответа на `login` или `create_user` клиент сам закроет соединение и переподключится с полученным UUID в `token`.

**Критерии приёмки:**

- [ ] Подключение с `token=` устанавливается без ошибки.
- [ ] Сообщения `login` и `create_user` обрабатываются.
- [ ] Сообщения других типов при пустом токене не приводят к выполнению логики чатов/сообщений.

---

### 3.2. Логин по username и password (P0)

**Цель:** аутентификация пользователя по учётным данным и выдача UUID для дальнейшей работы.

**Вход (от клиента):**

```json
{
  "type": "login",
  "content": "{\"username\":\"john_doe\",\"password\":\"secret123\"}"
}
```

- `content` — JSON-строка с полями `username` (строка), `password` (строка).

**Выход при успехе:**

```json
{
  "type": "login_success",
  "id": "uuid-пользователя"
}
```

- `id` — UUID пользователя из БД (тот же, что используется как токен).

**Выход при ошибке:**

```json
{
  "type": "error",
  "error": "Неверный логин или пароль"
}
```

**Требования:**

1. Парсить `content` как JSON, извлечь `username`, `password`.
2. Найти пользователя по `username` (учёт регистра, если задано в БД).
3. Проверить пароль (сравнение с хранимым хешем).
4. При совпадении — отправить `login_success` с `id` = UUID пользователя.
5. При несовпадении или отсутствии пользователя — отправить `error` с текстом, например: «Неверный логин или пароль».

**Критерии приёмки:**

- [ ] При верных username и password возвращается `login_success` с корректным `id`.
- [ ] При неверном пароле или несуществующем пользователе возвращается `error`.
- [ ] Сообщение обрабатывается только при подключении с пустым токеном (см. п. 3.1).

---

### 3.3. Регистрация пользователя (P2 — проверить)

**Цель:** убедиться, что ответ бэкенда совместим с клиентом.

**Вход:** уже реализован на бэкенде (`create_user`, `content` с `username`, `email`, `password`).

**Требование к ответу при успехе:**

```json
{
  "type": "user_created",
  "id": "uuid-созданного-пользователя"
}
```

- Поле **`id`** обязательно. Клиент сохраняет его как токен и использует для последующих подключений.

**Валидация (рекомендуется):**

- username: 3–50 символов;
- email: валидный формат;
- password: минимум 6 символов.

При нарушении — ответ `type: "error"` с полем `error`.

**Критерии приёмки:**

- [ ] В ответе на успешное создание пользователя всегда присутствует `id` (UUID).
- [ ] При ошибке валидации или дубликате возвращается `error` с понятным текстом.

---

### 3.4. Список чатов (P0)

**Цель:** клиент получает список чатов текущего пользователя с отображаемыми именами.

**Вход:**

```json
{
  "type": "get_chats"
}
```

- Отправляется только при подключении с валидным токеном (UUID пользователя).

**Выход:**

```json
{
  "type": "chats_list",
  "content": "{\"chats\":[{\"id\":\"...\",\"name\":\"...\",\"type\":\"private\",\"createdAt\":1234567890,\"updatedAt\":1234567890,\"lastMessageId\":\"...\",\"lastMessageTime\":1234567890}]}"
}
```

- `content` — JSON-строка.
- Внутри: объект с массивом `chats`.
- Каждый элемент чата:
  - `id` (string) — UUID чата;
  - `name` (string | null) — отображаемое имя;
  - `type` (string) — `"private"` или `"group"`;
  - `createdAt`, `updatedAt` (number) — Unix timestamp в миллисекундах;
  - `lastMessageId` (string | null), `lastMessageTime` (number | null).

**Требование по отображению:**

- Для чатов с `type: "private"` в поле **`name`** передавать **username второго участника** (не UUID и не null, если возможно). Иначе в интерфейсе в одном месте будет username, в другом — «цифры».

**Критерии приёмки:**

- [ ] По токену (UUID пользователя) возвращаются только чаты, где он участник.
- [ ] Формат ответа соответствует приведённому выше.
- [ ] Для личных чатов в `name` подставляется username второго участника.

---

### 3.5. Поиск пользователя по username (P1)

**Цель:** проверка существования пользователя перед открытием чата «Написать» (поведение как в Telegram).

**Вход:**

```json
{
  "type": "find_user",
  "content": "username_пользователя"
}
```

- `content` — **строка** (один username), не JSON-объект.

**Выход при найденном пользователе:**

```json
{
  "type": "user_found",
  "content": "{\"id\":\"uuid\",\"username\":\"john_doe\"}"
}
```

- `content` — JSON-строка с минимум полями `id`, `username` (для отображения «✓ @username» на клиенте).

**Выход при ненайденном:**

```json
{
  "type": "error",
  "error": "User not found"
}
```

**Требования:**

1. Поиск пользователя по полю username (без учёта регистра, если так задано в БД).
2. Не возвращать чувствительные данные (пароль, хеш и т.п.).

**Критерии приёмки:**

- [ ] При существующем username возвращается `user_found` с `id` и `username` в `content`.
- [ ] При несуществующем username возвращается `error` с текстом вида «User not found».

---

### 3.6. История сообщений (P0)

**Цель:** получение клиентом истории сообщений чата с пагинацией.

**Вход:**

```json
{
  "type": "get_messages",
  "chatId": "uuid-чата",
  "content": "{\"limit\":50,\"offset\":0}"
}
```

или:

```json
{
  "content": "{\"afterSequence\":100}"
}
```

- `chatId` — UUID чата.
- `content` — опциональная JSON-строка с параметрами:
  - `limit` (по умолчанию 50), `offset` (по умолчанию 0);
  - или `afterSequence` — вернуть сообщения с `sequence_number` больше указанного.

**Выход:**

```json
{
  "type": "messages_list",
  "chatId": "uuid-чата",
  "content": "{\"messages\":[{...}]}"
}
```

Структура элемента в `messages`:

- `id`, `chatId`, `senderId`, `content`, `clientMessageId`, `sequenceNumber`, `createdAt`, `updatedAt`, `status`, `isDeleted`.

**Требования:**

1. Проверять, что пользователь (по токену) является участником чата `chatId`.
2. Сообщения с `isDeleted: true` по необходимости исключать или помечать; клиент их не отображает.
3. Сортировка по `sequence_number` или `createdAt` (порядок — от старых к новым в массиве).

**Критерии приёмки:**

- [ ] Для участника чата возвращается `messages_list` с массивом сообщений в описанном формате.
- [ ] При отсутствии доступа к чату — ответ `error`.
- [ ] Поддержаны параметры `limit`, `offset` и при необходимости `afterSequence`.

---

### 3.7. Отправка сообщения в существующий чат (P2 — проверить)

**Цель:** идемпотентная отправка и подтверждение доставки.

**Вход:**

```json
{
  "type": "message",
  "chatId": "uuid-чата",
  "content": "Текст сообщения",
  "clientMessageId": "uuid-сообщения-с-клиента"
}
```

**Выход (ACK):**

```json
{
  "type": "ack",
  "id": "uuid-сообщения-на-сервере",
  "clientMessageId": "uuid-сообщения-с-клиента"
}
```

**Требования:**

1. Дедупликация по `clientMessageId` в рамках чата: при повторной отправке с тем же `chatId` и `clientMessageId` возвращать ACK с уже существующим `id`, не создавая дубликат.
2. Проверка прав: отправитель — участник чата.

**Критерии приёмки:**

- [ ] После сохранения сообщения отправляется ACK с `id` и `clientMessageId`.
- [ ] Повторный запрос с тем же `clientMessageId` и `chatId` возвращает тот же `id` без нового сообщения в БД.

---

### 3.8. Первое сообщение пользователю по username (P0)

**Цель:** создание личного чата «на лету» при первом сообщении по username и привязка чата на клиенте через `chatId` в ACK.

**Вход:**

```json
{
  "type": "message_to_user",
  "content": "{\"recipientUsername\":\"johndoe\",\"content\":\"Текст сообщения\"}",
  "clientMessageId": "uuid-сообщения-с-клиента"
}
```

- `content` — JSON-строка: `recipientUsername` (string), `content` (string) — текст сообщения.

**Требуемая логика:**

1. Определить текущего пользователя по токену (UUID).
2. Найти пользователя по `recipientUsername`.
3. Найти существующий личный чат между двумя пользователями или создать новый.
4. Сохранить сообщение в этот чат (с учётом дедупликации по `clientMessageId`, если применимо).
5. Отправить ACK с **обязательным** полем **`chatId`**:

```json
{
  "type": "ack",
  "id": "uuid-сообщения-на-сервере",
  "clientMessageId": "uuid-сообщения-с-клиента",
  "chatId": "uuid-чата"
}
```

6. Разослать получателям полное сообщение (тип `message` с полными данными) по текущей логике сервера.

**При ошибке (пользователь не найден и т.п.):**

```json
{
  "type": "error",
  "error": "User not found"
}
```

**Критерии приёмки:**

- [ ] При валидном `recipientUsername` создаётся/находится личный чат и сохраняется сообщение.
- [ ] В ACK всегда присутствует поле `chatId` с UUID этого чата.
- [ ] При несуществующем получателе возвращается `error`, сообщение не сохраняется.

---

### 3.9. Получение/создание личного чата по username (P3, опционально)

**Цель:** открыть чат с пользователем по username до отправки первого сообщения (если нужен такой сценарий).

**Вход:**

```json
{
  "type": "get_or_create_private_chat",
  "content": "{\"username\":\"johndoe\"}"
}
```

**Выход при успехе:**

```json
{
  "type": "chat_created",
  "id": "uuid-чата"
}
```

или

```json
{
  "type": "chat",
  "chatId": "uuid-чата"
}
```

Клиент принимает идентификатор чата из поля `id` или `chatId`.

**Критерии приёмки:**

- [ ] По `username` находится пользователь, находится или создаётся личный чат.
- [ ] В ответе возвращается идентификатор чата в поле `id` или `chatId`.

---

## 4. Общие требования к WebSocket

1. **Ping/Pong:** сервер шлёт `{"type":"ping"}`, клиент отвечает `{"type":"pong"}`. При отсутствии ответа допускается закрытие соединения.
2. **Ошибки:** любой сценарий ошибки должен возвращать сообщение вида `{"type":"error","error":"текст"}`.
3. **Формат:** все сообщения — JSON; числа времени — Unix timestamp в **миллисекундах**.
