# Требования бэкенда для веб-клиента CosaNostra

Документ описывает, **что отправляет клиент**, **какие ответы ожидает** и **что нужно доработать на бэкенде** для работы с веб-клиентом (логин, регистрация, чаты, сообщения).

---

## 1. Подключение по WebSocket

### Текущее поведение клиента

- **Обычное подключение (после входа):**  
  `ws://localhost:8080/chat?token=<UUID>`  
  В качестве `token` передаётся UUID пользователя (после логина или регистрации).

- **Подключение для входа/регистрации:**  
  Клиент подключается с **пустым токеном**:  
  `ws://localhost:8080/chat?token=`  
  и отправляет только сообщения типов `login` или `create_user`. После ответа соединение закрывается, клиент переподключается уже с полученным UUID.

### Что нужно на бэкенде

| Требование | Описание |
|------------|----------|
| Принимать пустой токен для auth | Разрешить подключение с `token=` (или отсутствующим token) **только** для обработки `login` и `create_user`. Остальные типы сообщений для таких сессий отклонять или игнорировать. |
| Токен = UUID пользователя | В MVP токен — это UUID пользователя. В ответах на `login` и `create_user` возвращать этот UUID в поле `id`. |

## 2. Регистрация пользователей

### Что отправляет клиент

```json
{
  "type": "create_user",
  "content": "{\"username\":\"john_doe\",\"email\":\"john@example.com\",\"password\":\"secret123\"}"
}
```

- `content` — JSON-строка с полями: `username`, `email`, `password`.

### Что ожидает клиент

**Успех:**

```json
{
  "type": "user_created",
  "id": "uuid-созданного-пользователя"
}
```

- Поле **`id`** обязательно — клиент сохраняет его как токен и использует для последующего входа и подключения к WebSocket.

**Ошибка:**

```json
{
  "type": "error",
  "error": "Текст ошибки (например: Username already exists)"
}
```

### Что нужно на бэкенде

| Статус | Действие |
|--------|----------|
| Уже есть | Убедиться, что в ответе на успешное создание пользователя всегда приходит `type: "user_created"` и **`id`** (UUID пользователя). |
| Проверить | Валидация: username 3–50 символов, email — валидный формат, password — минимум 6 символов; при нарушении — ответ `type: "error"` с понятным `error`. |

## 3. Вход (логин)

### Что отправляет клиент

```json
{
  "type": "login",
  "content": "{\"username\":\"john_doe\",\"password\":\"secret123\"}"
}
```

- Отправляется при подключении с **пустым токеном** (см. п. 1).

### Что ожидает клиент

**Успех:**

```json
{
  "type": "login_success",
  "id": "uuid-пользователя"
}
```

- **`id`** — UUID пользователя; клиент сохраняет его как токен и дальше подключается к WebSocket с этим токеном.

**Ошибка:**

```json
{
  "type": "error",
  "error": "Неверный логин или пароль"
}
```

### Что нужно на бэкенде

| Статус | Действие |
|--------|----------|
| Нужно реализовать | Обработка сообщения `type: "login"`. |
| | Парсинг `content` как JSON с полями `username`, `password`. |
| | Проверка учётных данных (например, по хешу пароля в БД). |
| | При успехе — ответ `type: "login_success"`, **`id`** = UUID пользователя. |
| | При неуспехе — ответ `type: "error"` с полем `error`. |

Без реализации логина пользователь может войти только по токену (UUID) после регистрации (кнопка «Войти по токену» на экране входа).

## 4. Чаты

### 4.1 Получение списка чатов

**Запрос клиента:**

```json
{
  "type": "get_chats"
}
```

**Ожидаемый ответ:**

```json
{
  "type": "chats_list",
  "content": "{\"chats\":[{\"id\":\"uuid\",\"name\":\"Chat Name\",\"type\":\"private\",\"createdAt\":1234567890,\"updatedAt\":1234567890,\"lastMessageId\":\"uuid\",\"lastMessageTime\":1234567890}]}"
}
```

- `content` — JSON-строка с объектом `{ "chats": Chat[] }`.
- Клиент парсит и ожидает у каждого чата поля: `id`, `name` (или null), `type` (`"private"` | `"group"`), `createdAt`, `updatedAt`, `lastMessageId`, `lastMessageTime` (числа — Unix timestamp в миллисекундах).
- **Важно для отображения:** для личных чатов (`type: "private"`) в поле **`name`** должен приходить **username второго участника**, чтобы в интерфейсе везде отображалось имя пользователя, а не UUID (иначе в одном месте видно username, в другом — «цифры»).

### 4.2 Поиск пользователя по username (find_user)

Клиент перед открытием чата «Написать» проверяет существование пользователя (как в Telegram — нельзя написать несуществующему).

**Запрос клиента:**

```json
{
  "type": "find_user",
  "content": "username_пользователя"
}
```

- `content` — строка с username (не JSON-объект).

**Ожидаемый ответ (найдены похожие пользователи, топ-5):**

```json
{
  "type": "user_found",
  "content": "{\"users\":[{\"id\":\"uuid1\",\"username\":\"testcn\"},{\"id\":\"uuid2\",\"username\":\"testcase\"}]}"
}
```

или массив напрямую:

```json
{
  "type": "user_found",
  "content": "[{\"id\":\"uuid1\",\"username\":\"testcn\"},{\"id\":\"uuid2\",\"username\":\"testcase\"}]"
}
```

- Клиент выводит **все** похожие пользователи списком (✓ @username). Каждый элемент кликабелен — открывается чат с этим пользователем.
- Поддерживается и один объект `{ "id", "username" }` в `content` (без массива) для обратной совместимости.

**Ожидаемый ответ (не найден):**

```json
{
  "type": "error",
  "error": "User not found"
}
```

Без реализации `find_user` клиент не сможет показать проверку «существует ли пользователь» и будет открывать чат без проверки.

### 4.3 Получение или создание личного чата по username (опционально)

**Запрос клиента:**

```json
{
  "type": "get_or_create_private_chat",
  "content": "{\"username\":\"johndoe\"}"
}
```

**Ожидаемый ответ (успех):**

```json
{
  "type": "chat_created",
  "id": "uuid-чата"
}
```

или

```json
{
  "type": "chat",
  "chatId": "uuid-чата"
}
```

- Клиент принимает идентификатор чата из поля **`id`** или **`chatId`** и открывает этот чат.

### 4.4 Что нужно на бэкенде по чатам

| Функция | Статус | Действие |
|---------|--------|----------|
| `get_chats` | Реализовать/проверить | По токену (UUID пользователя) вернуть список чатов, где он участник. Формат ответа — `chats_list`. Для личных чатов в **`name`** передавать **username** второго участника. |
| `find_user` | Рекомендуется | По `content` (username) проверить существование пользователя. Ответ `user_found` с `content` (JSON с `username`) или `error` «User not found». |
| `get_or_create_private_chat` | Опционально | По `username` из `content` найти пользователя, найти или создать личный чат между текущим пользователем и ним, вернуть `chat_created` или `chat` с `id` или `chatId`. |

## 5. Сообщения

### 5.1 Отправка сообщения в существующий чат

**Запрос клиента:**

```json
{
  "type": "message",
  "chatId": "uuid-чата",
  "content": "Текст сообщения",
  "clientMessageId": "uuid-сообщения-с-клиента"
}
```

**Ожидаемый ответ (ACK):**

```json
{
  "type": "ack",
  "id": "uuid-сообщения-на-сервере",
  "clientMessageId": "uuid-сообщения-с-клиента"
}
```

- Дедупликация по `clientMessageId`: при повторной отправке с тем же `clientMessageId` в том же чате — вернуть ACK с уже существующим `id` (идемпотентность).

### 5.2 Первое сообщение пользователю по username (создание чата)

**Запрос клиента:**

```json
{
  "type": "message_to_user",
  "content": "{\"recipientUsername\":\"johndoe\",\"content\":\"Текст сообщения\"}",
  "clientMessageId": "uuid-сообщения-с-клиента"
}
```

**Ожидаемое поведение бэкенда:**

1. Найти пользователя по `recipientUsername`.
2. Найти или создать личный чат между текущим пользователем (из токена) и получателем.
3. Сохранить сообщение в этот чат.
4. Отправить **ACK** с обязательным полем **`chatId`**, чтобы клиент добавил чат в список и привязал к нему переписку:

```json
{
  "type": "ack",
  "id": "uuid-сообщения-на-сервере",
  "clientMessageId": "uuid-сообщения-с-клиента",
  "chatId": "uuid-чата"
}
```

- Рассылка сообщения отправителю и получателю (тип `message` с полными данными) — по текущей логике сервера.

### 5.3 Что нужно на бэкенде по сообщениям

| Функция | Статус | Действие |
|---------|--------|----------|
| `message` (с `chatId`) | Реализовано/проверить | ACK с `id` и `clientMessageId`; дедупликация по `clientMessageId` в рамках чата. |
| `message_to_user` | Реализовать | Обработка `recipientUsername` + текст из `content`, создание/поиск личного чата, сохранение сообщения, **обязательный `chatId` в ACK**. |

## 6. История сообщений

**Запрос клиента:**

```json
{
  "type": "get_messages",
  "chatId": "uuid-чата",
  "content": "{\"limit\":50,\"offset\":0}"
}
```

или с пагинацией по `sequence_number`:

```json
{
  "type": "get_messages",
  "chatId": "uuid-чата",
  "content": "{\"afterSequence\":100}"
}
```

**Ожидаемый ответ:**

```json
{
  "type": "messages_list",
  "chatId": "uuid-чата",
  "content": "{\"messages\":[{\"id\":\"uuid\",\"chatId\":\"uuid\",\"senderId\":\"uuid\",\"content\":\"Текст\",\"clientMessageId\":\"uuid\",\"sequenceNumber\":1,\"createdAt\":1234567890,\"updatedAt\":1234567890,\"status\":\"sent\",\"isDeleted\":false}]}"
}
```

- Клиент ожидает в каждом сообщении: `id`, `chatId`, `senderId`, `content`, `clientMessageId`, `sequenceNumber`, `createdAt`, `updatedAt`, `status`, `isDeleted`. Сообщения с `isDeleted: true` клиент не отображает.

### Что нужно на бэкенде

| Статус | Действие |
|--------|----------|
| Реализовать/проверить | Обработка `get_messages` с `chatId` и опциональными параметрами в `content` (`limit`, `offset`, `afterSequence`). |
| | Ответ `messages_list` с массивом сообщений в описанном формате; проверка, что пользователь имеет доступ к чату. |

---

## 7. Прочее (уже используется клиентом)

- **Ping/Pong:** сервер шлёт `{"type":"ping"}`, клиент отвечает `{"type":"pong"}`.
- **Ошибки:** клиент обрабатывает `{"type":"error","error":"текст"}` и при необходимости показывает их пользователю (в т.ч. при ошибках, связанных с `clientMessageId`).

## 8. Сводная таблица доработок бэкенда

| № | Функция | Приоритет | Кратко |
|---|---------|-----------|--------|
| 1 | Пустой токен для auth | Высокий | Разрешить подключение с пустым token только для `login` и `create_user`. |
| 2 | Логин по username + password | Высокий | Обработка `login`, ответ `login_success` с `id` (UUID пользователя). |
| 3 | Регистрация | Проверить | Ответ `user_created` с полем `id`. |
| 4 | Список чатов | Высокий | Ответ на `get_chats` в формате `chats_list` (см. п. 4.1). |
| 5 | История сообщений | Высокий | Ответ на `get_messages` в формате `messages_list` (см. п. 6). |
| 6 | Сообщение в чат + ACK | Проверить | ACK с `id` и `clientMessageId`; дедупликация по `clientMessageId`. |
| 7 | Первое сообщение по username | Высокий | Обработка `message_to_user`, создание/поиск чата, ACK с **`chatId`**. |
| 8 | get_or_create_private_chat | Низкий | Опционально: ответ с `id` или `chatId` чата по username. |

После реализации п. 1–2 пользователь сможет входить по логину и паролю; после п. 4–7 — видеть чаты, историю и начинать переписку по username без ручного ввода UUID чата.
